## 2. One-line summary

A small OmniAuth OAuth 2.0 strategy gem for **TikTok Login Kit** (TikTok's developer OAuth flow at `developers.tiktok.com`). LFM forks the upstream `Lianowar/omniauth-tiktok` to update default scopes and the `raw_info` field list. Probably consumed by `lfm-accounts` (for TikTok login) or `cms`/`lfm-ingest` (for connecting a TikTok data source).

## 5. Auth

OAuth 2.0 Authorization Code flow against TikTok with non-standard parameter names:

- **Authorize URL**: `https://www.tiktok.com/v2/auth/authorize/`
- **Token URL**: `https://open.tiktokapis.com/v2/oauth/token/`
- **Default scope**: `user.info.basic,user.info.profile,user.info.stats,video.list` (LFM-customized — see §13.2)
- **Token grant_type**: `authorization_code` (standard)

⚠ **TikTok uses `client_key` instead of `client_id`** for both authorize and token requests. The strategy explicitly remaps:
```ruby
def authorize_params
  super.tap do |params|
    params[:scope] ||= DEFAULT_SCOPE
    params[:response_type] = 'code'
    params.delete(:client_id)
    params[:client_key] = options.client_id
  end
end

def token_params
  super.tap do |params|
    params.delete(:client_id)
    params[:client_key] = options.client_id
  end
end
```

A custom `extract_access_token` proc is also wired (in `client_options`) because TikTok returns the access_token in a non-standard JSON shape.

## 8. Data flow / lifecycle

```
Consumer mounts: provider :tiktok, ENV['TIKTOK_CLIENT_ID'], ENV['TIKTOK_CLIENT_SECRET']
   ↓
User clicks "Sign in with TikTok"
   ↓ GET /auth/tiktok
OmniAuth::Strategies::Tiktok#request_phase → builds authorize URL with `client_key=<TIKTOK_CLIENT_ID>` (NOT client_id)
   ↓ Redirects to https://www.tiktok.com/v2/auth/authorize/?client_key=...&scope=...&response_type=code
   ↓
User logs in, approves scopes
   ↓
TikTok redirects to /auth/tiktok/callback?code=...
   ↓
OmniAuth callback handler → POST https://open.tiktokapis.com/v2/oauth/token/ with `client_key=...&client_secret=...&grant_type=authorization_code&code=...`
   ↓ extract_access_token proc unwraps TikTok's response shape
   ↓ access_token.get('https://open.tiktokapis.com/v2/user/info/?fields=open_id,union_id,avatar_url,username,profile_deep_link&access_token=...')
   ↓ Build AuthHash:
     {
       provider: 'tiktok',
       uid: access_token.params['open_id'],
       info: { 'nickname' => raw_info['data']['display_name'] },
       extra: { raw_info: <fields> },
       credentials: { token, refresh_token, expires_at, expires: true, refresh_token_expires_at }
     }
   ↓ Consumer's callback handler reads request.env['omniauth.auth']
```

⚠ **Token lifetime**: TikTok access tokens are valid for **24 hours only** (per README). Refresh tokens are valid for 365 days. The strategy correctly populates `credentials.refresh_token_expires_at` from `Time.now.to_i + access_token.params['refresh_expires_in'].to_i`.

## 13. Conventions & gotchas

### 13.1 NO committed credentials

A grep across `lib/`, `spec/`, `Gemfile`, `.gemspec` for `ghp_*`, `AKIA*`, hardcoded passwords, OAuth secrets returned **zero hits**. The strategy correctly delegates `client_id` + `client_secret` to consumer-side env vars (`ENV['TIKTOK_CLIENT_ID']` / `ENV['TIKTOK_CLIENT_SECRET']`). **Not a finding.**

### 13.2 ⚠ TikTok uses `client_key`, NOT `client_id`

The strategy's `authorize_params` and `token_params` BOTH delete the standard OAuth2 `client_id` parameter and substitute `client_key`. This is TikTok's non-standard naming. Don't try to "fix" this — it's load-bearing for the TikTok OAuth flow.

### 13.3 ⚠ Custom `extract_access_token` proc

`option :client_options, { extract_access_token: proc do |client, hash| ... end }` — TikTok's token response is wrapped differently than standard OAuth2 expects. The proc unwraps `hash.delete('access_token')` (or `:access_token`) and constructs the `OAuth2::AccessToken` manually. If you bump `omniauth-oauth2` to a major version that changes this contract, update the proc.

### 13.4 24-hour access token lifetime

Per README: "Tiktok access_token valid only for 24 hours!" The strategy sets `credentials.expires = true` always (since TikTok always provides expiration). The refresh_token is valid for 365 days. Consumer code MUST refresh tokens daily using the stored `refresh_token` against the same `/v2/oauth/token/` endpoint with `grant_type=refresh_token`.

### 13.5 `info` block returns ONLY `nickname` from raw_info's `display_name`

```ruby
info do
  prune!('nickname' => raw_info['data']['display_name'])
end
```

So `request.env['omniauth.auth'][:info]` only contains `{ nickname: ... }`. If consumers want the full TikTok profile (avatar_url, username, profile_deep_link, etc.), they need to read `request.env['omniauth.auth'][:extra][:raw_info]`. _<TODO: consider expanding `info` to include avatar_url + username — most consumers will want them>_

### 13.6 `raw_info` URL hardcodes `?access_token=...` AS A QUERY PARAMETER

```ruby
@raw_info ||= access_token.get("https://open.tiktokapis.com/v2/user/info/?fields=...&access_token=#{access_token.token}").parsed
```

The access_token is sent as a URL query param (NOT as `Authorization: Bearer <token>` header). This is OK per TikTok's API spec but **logs the access_token in any HTTP intermediary that records URL paths** (CloudFront, ALB access logs, CDN logs, Sentry breadcrumbs). _<TODO: confirm whether TikTok v2 API actually requires the query param OR accepts the standard Authorization: Bearer header — if the latter, switch to header to avoid log leakage>_

### 13.7 `prune!` recursively strips nil/empty values

`prune!(hash)` walks the hash recursively and deletes keys whose values are nil or `.empty?`. Used to keep `info`, `extra`, `credentials` clean. Standard OmniAuth pattern.

## 17. AI agent guidance / do's and don'ts

### Do

- ✅ Follow TikTok's developer docs at `https://developers.tiktok.com/doc/login-kit-web` — TikTok's OAuth changes frequently.
- ✅ When extending the strategy, remember TikTok uses `client_key` (NOT `client_id`) on every request — `authorize_params` and `token_params` both remap it.
- ✅ When the consumer needs more user fields, modify the `raw_info` URL's `fields=` query (TikTok's `/v2/user/info/` endpoint requires explicit field list).
- ✅ When refreshing tokens, use `grant_type=refresh_token` against `https://open.tiktokapis.com/v2/oauth/token/` — store `refresh_token_expires_at` (365 days from issuance) so consumers know when to re-prompt.

### Don't

- ❌ Don't change `client_key` back to `client_id` — TikTok's token endpoint will return 400 invalid_client.
- ❌ Don't remove the `extract_access_token` proc in `client_options` — TikTok's response shape doesn't match standard OAuth2.
- ❌ Don't add new scopes without verifying TikTok's app dashboard has them enabled — TikTok requires per-scope app review.
- ❌ Don't commit `TIKTOK_CLIENT_ID` / `TIKTOK_CLIENT_SECRET` in source — they belong in env / Secrets Manager.
- ❌ Don't rely on `info[:nickname]` alone — the `info` block only exposes that one field; everything else is in `extra[:raw_info]`.
